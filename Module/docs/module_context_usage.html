ModuleContext 使用文档

1. 模块作用

ModuleContext 是模块系统的统一访问层，它提供了：

  • ModuleContext：非模板基类，提供统一的模块上下文访问接口，内部保存按全局槽位编号存储的指针数组
  • ModuleContextManager<Modules...>：模板化的模块上下文管理器，负责按模板参数初始化模块指针
  • GeneratedModuleContext：自动生成的模块上下文类型，使用预定义的模块列表

它是主程序和插件之间共享模块上下文的桥梁，既保留了模板类型安全，又保留了跨插件共享时最重要的固定槽位 ABI 语义。

2. 核心概念

2.1 全局槽位 (Fixed Slots)

每个模块类型在编译期都有固定的槽位索引，由 module_include_gen.h 定义：

Application::ApplicationModule  -> 槽位 0
Render::RenderModule            -> 槽位 1
Resource::ResourceManagerModule -> 槽位 2

2.2 ModuleManager

模块系统的唯一真实模块拥有者，负责：

  • 真正拥有模块实例
  • 启动、关闭模块
  • 建立槽位表
  • 给主程序和插件提供查询能力

2.3 ModuleContext

统一的模块访问容器，负责：

  • 缓存模块指针（不拥有对象）
  • 封装按类型获取模块的逻辑
  • 支持从 ModuleManager 和 ModuleHostAPI 初始化

3. 基本用法

3.1 在主程序中使用

#include "module_context.h"
#include "module_manager.h"

int main() {
    // 1. 启动 ModuleManager
    auto& manager = ModuleManager::Instance();
    if (!manager.Startup()) {
        return -1;
    }

    // 2. 创建并初始化 ModuleContext
    GeneratedModuleContext context;
    if (!context.Initialize(manager)) {
        return -1;
    }

    // 3. 通过上下文访问模块
    auto* app = context.Get<Application::ApplicationModule>();
    auto* render = context.Get<Render::RenderModule>();
    auto* resource = context.Get<Resource::ResourceManagerModule>();

    // 4. 使用模块...

    // 5. 清理
    manager.Shutdown();
    return 0;
}

3.2 在插件中使用

#include "module_context.h"

extern "C" bool PluginStartup(const ModuleHostAPI* host_api) {
    if (host_api == nullptr) {
        return false;
    }

    // 1. 创建并初始化 ModuleContext
    GeneratedModuleContext context;
    if (!context.Initialize(*host_api)) {
        return false;
    }

    // 2. 通过上下文访问模块
    auto* render = context.Get<Render::RenderModule>();
    if (render == nullptr) {
        return false;
    }

    // 3. 使用模块...

    return true;
}

4. API 详细说明

4.1 ModuleContext 基类

方法 说明
Reset() 重置所有槽位为空指针
IsBound(index) 检查指定槽位是否已绑定
Get<T>() 按类型获取模块指针，如果未绑定则返回 nullptr
GetRaw(index) 获取指定槽位的原始指针
SetSlot(index, ptr) 设置指定槽位的指针（protected）

4.2 ModuleContextManager 模板类

方法 说明
Initialize(manager) 从 ModuleManager 初始化上下文
Initialize(api) 从 ModuleHostAPI 初始化上下文

5. 完整示例

5.1 最小可用示例

#include "module_context.h"
#include "module_manager.h"
#include "application_module.h"
#include "render_module.h"

int main() {
    // 启动模块管理器
    auto& manager = ModuleManager::Instance();
    if (!manager.Startup()) {
        return -1;
    }

    // 创建上下文
    GeneratedModuleContext context;
    if (!context.Initialize(manager)) {
        manager.Shutdown();
        return -1;
    }

    // 获取模块
    auto* app = context.Get<Application::ApplicationModule>();
    auto* render = context.Get<Render::RenderModule>();

    if (!app || !render) {
        manager.Shutdown();
        return -1;
    }

    // 使用模块...
    // ...

    // 清理
    manager.Shutdown();
    return 0;
}

5.2 验证指针一致性

// 通过 ModuleContext 获取
auto* context_app = context.Get<Application::ApplicationModule>();

// 直接从 ModuleManager 获取
auto* direct_app = manager.GetModule<Application::ApplicationModule>();

// 验证两者是相同的指针
assert(context_app == direct_app);

5.3 检查模块绑定状态

ModuleContext context;
context.Reset();

// 检查槽位是否已绑定
bool appBound = context.IsBound(0);  // false
bool renderBound = context.IsBound(1); // false

// 初始化后
context.Initialize(manager);

// 再次检查
appBound = context.IsBound(0);  // true
renderBound = context.IsBound(1); // true

6. ModuleHostAPI 初始化

ModuleHostAPI 是跨 DLL / 跨 CRT 边界时共享模块的 C ABI 风格接口：

6.1 ModuleHostAPI 结构

struct ModuleHostAPI {
    void* context = nullptr;
    void* (*query_by_index)(void* context, std::size_t index) noexcept = nullptr;
};

6.2 从 ModuleManager 获取 HostAPI

const ModuleHostAPI& hostAPI = manager.GetHostAPI();

6.3 使用 HostAPI 初始化上下文

GeneratedModuleContext context;
if (!context.Initialize(hostAPI)) {
    return false;
}

// 现在可以像平常一样使用
auto* render = context.Get<Render::RenderModule>();

7. 自定义模块上下文

除了使用 GeneratedModuleContext，你还可以创建自定义的模块上下文：

7.1 定义自定义上下文类型

using MyModuleContext = ModuleContextManager<
    Application::ApplicationModule,
    Render::RenderModule
>;

7.2 使用自定义上下文

MyModuleContext context;
if (!context.Initialize(manager)) {
    return false;
}

// 只包含声明的模块类型
auto* app = context.Get<Application::ApplicationModule>();
auto* render = context.Get<Render::RenderModule>();
// auto* resource = context.Get<Resource::ResourceManagerModule>(); // 编译错误

8. 模块上下文生成

module_context_gen.h 负责生成当前上下文所需模块类型列表：

8.1 默认生成列表

#define MODULE_CONTEXT_GEN_TYPES \
    Application::ApplicationModule, \
    Render::RenderModule, \
    Resource::ResourceManagerModule

using GeneratedModuleContext = ModuleContextManager<MODULE_CONTEXT_GEN_TYPES>;

8.2 修改生成列表

如需修改生成的模块列表，编辑 module_context_gen.h 中的 MODULE_CONTEXT_GEN_TYPES 宏定义。

注意：修改 module_context_gen.h 后，所有使用 GeneratedModuleContext 的地方将自动同步。

10. 注意事项

1. 生命周期管理： ModuleContext 不拥有模块对象，只缓存模块指针。模块的生命周期由 ModuleManager 管理。
2. 线程安全： 当前实现不提供线程安全保证。如需在多线程环境中使用，请自行添加同步机制。
3. 初始化顺序： 确保在使用 ModuleContext 之前，ModuleManager 已经启动。
提示： 推荐始终使用 GeneratedModuleContext，这样可以确保主程序和插件使用相同的模块列表。

11. 常见问题

Q1: 为什么 Get<T>() 返回 nullptr？

可能原因：

  • ModuleManager 未正确启动
  • ModuleContext 未正确初始化
  • 模块类型 T 未在 module_include_gen.h 中注册

Q2: 可以在插件中直接使用 ModuleManager 吗？

不推荐。插件应该通过 ModuleHostAPI 和 ModuleContext 访问模块，这样可以确保跨 DLL / 跨 CRT 兼容性。

Q3: 如何添加新的模块类型？

  1. 在 module_include_gen.h 中注册新模块类型和槽位
  2. 在 module_context_gen.h 中添加到 MODULE_CONTEXT_GEN_TYPES 宏
  3. 更新 ModuleManager::BuildModuleTable() 以包含新模块

12. 总结

ModuleContext 提供了统一的模块访问层，使得主程序和插件能够以相同的方式访问模块。通过固定槽位和类型安全的模板接口，它既保证了跨 DLL / 跨 CRT 兼容性，又提供了编译期类型检查。
