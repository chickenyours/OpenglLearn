Render 模块架构文档

1. 模块概述

Render 模块是图形渲染的核心模块，提供了跨图形 API 的 RHI（Render Hardware Interface）抽象层。它采用前端命令代理 + 专职渲染线程 + 委托后门的架构设计，实现了渲染操作与图形 API 的解耦。

设计目标：
  • 跨图形 API 抽象（当前支持 OpenGL 4.6+）
  • 命令代理执行，支持未来扩展多后端
  • 专职渲染线程，保证图形 API 调用的线程安全性
  • 委托后门，支持快速验证和过渡功能

2. 架构分层

+------------------+ +------------------+ +------------------+ | 上层模块/应用 | --> | RHI 前端层 | --> | 命令层 | | (Application/ECS)| | (RHIFrontend) | | (RHICommand) | +------------------+ | - CPU 资源加载 | | - 命令投递 | | - 命令组装 | +------------------+ +------------------+ | v +------------------+ +------------------+ +------------------+ | OpenGL/D3D/VK | <-- | 后端设备层 | <-- | 命令队列 | | 图形 API | | (Device) | | (CommandQueue) | +------------------+ | - GPU 资源创建 | +------------------+ | - GPU 数据上传 | | | - 图形 API 调用 | v +------------------+ +------------------+ | 渲染线程 | | (RenderThread) | +------------------+

2.1 RHI 前端层（RHIFrontend）

给上层模块调用的统一接口，负责CPU 端操作：

  • 资源加载：从文件/内存加载纹理资产（TextureAsset）到 CPU 内存
  • 接收资源请求：创建/更新/销毁缓冲、输入布局等
  • 分配逻辑句柄（Handle）
  • 组装 RHI 命令
  • 投递命令到命令队列
  • 不直接执行图形 API

2.2 命令层（RHICommand）

统一表示所有 RHI 操作，负责：

  • 定义命令类型（枚举）
  • 定义命令负载（Payload）
  • 进入统一队列
  • 保证提交顺序 = 执行顺序

2.3 后端设备层（Device）

后端设备抽象接口，负责GPU 端操作：

  • 创建 GPU 纹理资源：从 TextureAsset 创建 RHITexture
  • 上传纹理数据到 GPU：将像素数据从 CPU 内存上传到显存
  • 创建/销毁 GPU 缓冲区
  • 绑定缓冲区到渲染管线
  • 创建/销毁输入布局
  • 直接调用图形 API（OpenGL/D3D/VK）

2.4 渲染线程（RenderThread）

专职渲染线程，负责：

  • 创建和管理线程生命周期
  • 持有图形上下文
  • 维护运行循环（等待 - 批量执行）
  • 调用后端执行器执行命令

3. 核心组件

3.1 RenderModule

模块入口类，继承自 IModule 接口。

职责 管理图形库平台状态、OpenGL 函数加载、RHI 设备创建
关键方法 Startup(), Shutdown(), GetDevice(), InitializeFromApplication()
状态机 Uninitialized → PlatformReady → ContextReady → FunctionLoaderReady → Ready

3.2 RHI::Device（后端设备）

RHI 后端设备抽象，只负责 GPU 端操作。

职责 创建 GPU 纹理资源、上传纹理数据、创建/销毁 GPU 缓冲区、绑定操作
关键方法 CreateTextureFromAsset(), UploadTexture(), CreateBuffer(), BindVertexBuffer()
实现 OpenGL 后端通过 CreateOpenGLDevice() 创建
注意 不负责 CPU 端资源加载（如 TextureAsset 加载），那是 RHIFrontend 的职责

3.3 RHI::RHIFrontend（前端接口）

RHI 前端接口，负责 CPU 端操作和命令投递。

职责 CPU 端资源加载、分配句柄、组装命令、投递到队列
关键方法 LoadTextureFromFile() (CPU 加载), CreateBuffer(), CreateTextureFromAsset(), UploadTexture(), EnqueueDelegate()
线程模型 可在任意线程调用，命令入队后唤醒渲染线程

3.4 RHI::RenderThread

专职渲染线程。

职责 持有图形上下文、维护运行循环、批量执行命令
关键方法 Start(), Stop(), Enqueue(), DrainOnce()
内部组件 RHICommandQueue, IRHIBackendExecutor

3.5 RHI::RHICommandQueue

统一命令队列。

职责 线程安全的 FIFO 队列，批量出队
关键方法 Push(), PopAll(), Empty()
线程安全 使用互斥锁保证多线程入队安全

3.6 IRHIBackendExecutor

后端命令执行器接口。

职责 接收命令、操作资源表、调用图形 API、执行委托
关键方法 Initialize(), Shutdown(), Execute(), GetDelegateContext()
OpenGL 实现 GLBackendExecutor

3.7 RHIBackendResourceTable

后端资源表（仅渲染线程内部使用）。

职责 逻辑句柄到真实后端对象的映射
关键方法 RegisterBuffer(), FindBuffer(), RemoveBuffer()
线程模型 仅渲染线程访问，无需同步

4. 资源类型系统

4.1 纹理系统（职责分离）

TextureAsset (CPU 端资源) RHITexture (GPU 端资源) +----------------------+ +---------------------+ | - 文件加载 (CPU) | | - GPU 纹理描述 | | - 原始像素数据 (CPU) | -------> | - ResourceState | | - TextureDesc | 创建时 | - GetNativeHandle() | | | | - 后端具体实现 | +----------------------+ +---------------------+ ^ ^ | | RHIFrontend Device 负责加载 负责创建和上传
  • TextureAsset（CPU 端）：
    • 通过 RHIFrontend::LoadTextureFromFile() 加载
    • 存储图像文件加载后的原始像素数据（CPU 内存）
    • 包含纹理描述（TextureDesc）
    • 不表示 GPU 资源
  • RHITexture（GPU 端）：
    • 通过 Device::CreateTextureFromAsset() 创建
    • 代表 GPU 纹理资源（OpenGL Texture Object）
    • 包含资源状态管理和原生句柄接口
    • 通过 Device::UploadTexture() 接收数据
职责划分：
  • RHIFrontend：负责 CPU 端加载（LoadTextureFromFile 返回 TextureAsset）
  • Device：负责 GPU 端操作（创建 GPU 纹理、上传数据）

4.2 缓冲系统

类型 用途 Usage 标志
Vertex Buffer 存储顶点数据 BufferUsage::Vertex
Index Buffer 存储索引数据 BufferUsage::Index
Constant Buffer 存储 Uniform 数据 BufferUsage::Constant
Storage Buffer Shader 读写数据 BufferUsage::Storage
Transfer Buffer 数据拷贝中转 BufferUsage::TransferSrc/TransferDst

4.3 输入布局系统

管理顶点输入布局（VAO），定义顶点属性格式和绑定。

  • VertexAttributeDesc：顶点属性描述（location、binding、format、offset）
  • VertexBufferBindingDesc：顶点缓冲绑定描述（binding、stride、inputRate）
  • InputLayoutDesc：完整的输入布局描述

// 命令负载示例

    BufferDesc desc;

    BackendDelegate delegate;  // std::function<void(BackendDelegateContext&)>

using RHICommandPayload = std::variant<...>;

    RHICommandType type;

5.3 委托后门

委托后门是过渡验证通道，允许在渲染线程直接执行图形 API 调用。

// 定义委托函数类型
using BackendDelegate = std::function<void(BackendDelegateContext&)>;

// 委托上下文
struct BackendDelegateContext {
    void* backendExecutor = nullptr;  // 后端执行器指针
    void* nativeDevice = nullptr;      // 原生设备指针
    void* nativeContext = nullptr;     // 原生上下文指针
};

// 使用示例
frontend.EnqueueDelegate([](BackendDelegateContext& ctx) {
    glClearColor(0.2f, 0.3f, 0.4f, 1.0f);
    glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);
});
委托后门限制：
  1. 只能在渲染线程执行
  2. 仅作为调试/验证/过渡机制
  3. 不能让上层长期依赖
  4. 新增正式 RHI 能力后应逐步替换

6. 处理机制

6.1 纹理加载流程（CPU/GPU 职责分离）

步骤 1: CPU 端加载 (RHIFrontend) 业务线程 --LoadTextureFromFile()> RHIFrontend --读取文件> --加载像素数据到 CPU 内存> --创建 TextureAsset> 返回 asset 步骤 2: GPU 端创建 (Device) 业务线程 --CreateTextureFromAsset(asset)> Device --创建 OpenGL Texture 对象> --分配 GPU 显存> --返回 RHITexture*> 完成 步骤 3: GPU 端上传 (Device) 业务线程 --UploadTexture(texture, asset)> Device --从 asset 读取像素数据> --glTextureSubImage2D> --数据上传到 GPU> 完成

6.2 命令提交流程

业务线程 渲染线程 | | | 1. RHIFrontend::CreateBuffer() | | | | 2. 分配 BufferHandle | | | | 3. 创建 CreateBufferCmd | | - 复制 desc 数据 | | - 复制 initialData | | | | 4. RHICommandQueue::Push() | | (互斥锁保护) | | | | 5. 唤醒渲染线程 (条件变量) | | |----------> | | 6. 等待条件变量 | | | | 7. RHICommandQueue::PopAll() | | (批量取命令) | | | | 8. GLBackendExecutor::Execute() | | - 查资源表 | | - 调用 glCreateBuffers() | | - 调用 glNamedBufferData() | | | | 9. 登记到 RHIBackendResourceTable

6.2 渲染线程循环

void RenderThread::ThreadMain() {
    while (!stopRequested_) {
        // 1. 等待命令（条件变量）
        std::unique_lock<std::mutex> lock(mutex_);
        cv_.wait(lock, [this] {
            return stopRequested_ || !queue_.Empty();
        });

        if (stopRequested_) {
            break;
        }

        // 2. 批量取命令
        queue_.PopAll(commandBuffer_);

        // 3. 顺序执行命令
        for (const auto& cmd : commandBuffer_) {
            executor_->Execute(cmd);
        }

        commandBuffer_.clear();
    }
}

6.3 资源生命周期

创建流程: 业务线程 --CreateBuffer()--> RHIFrontend --分配 Handle> --创建命令> --入队> --唤醒> 渲染线程 --执行> --glCreateBuffers> --登记资源表> 完成 销毁流程: 业务线程 --DestroyBuffer(handle)> RHIFrontend --创建 Destroy 命令> --入队> --唤醒> 渲染线程 --查资源表> --glDeleteBuffers> --移除资源表> 完成

7. 线程模型

7.1 线程职责

线程 职责 可调用接口
主线程/业务线程 逻辑处理、命令提交 RHIFrontend 所有接口
渲染线程 图形 API 调用、资源管理 IRHIBackendExecutor、OpenGL API

7.2 同步机制

  • 命令队列：使用互斥锁保证多线程入队安全
  • 唤醒机制：条件变量，队列从空变非空时唤醒
  • 资源表：仅渲染线程访问，无需同步
  • 停止机制：标志位 + 条件变量唤醒

7.3 线程安全保证

组件 线程安全 说明
RHIFrontend 是（命令入队） 内部使用互斥锁
RHICommandQueue 是 FIFO 队列，互斥锁保护
RenderThread 是（启动/停止） 条件变量同步
RHIBackendResourceTable 否 仅渲染线程访问

10. 扩展方向

10.1 多后端支持

当前架构支持未来扩展多图形 API 后端：

  • D3D11/D3D12 后端：实现 IRHIBackendExecutor 接口
  • Vulkan 后端：使用 VkImage、VkBuffer、VkDescriptorSet
  • Metal 后端：使用 MTLTexture、MTLBuffer

10.2 未来优化方向

  • 多队列调度：分离渲染管线队列和资源上传队列
  • 命令重排：优化状态切换，减少冗余调用
  • 并行翻译：多线程命令翻译，单线程提交
  • Frame Graph：显式管理渲染依赖和资源生命周期
  • 持久映射：支持 persistent mapped buffers
第一阶段目标：
把 RHI 从"直接执行"改成"前端发命令，渲染线程顺序代理执行"，并保留一个能直接写图形库 API 的委托后门。

11. 双队列架构

11.1 设计目的

双队列架构将资源创建和资源删除操作分离到两个独立的队列中，主要目的是：

  • 避免资源悬挂引用：先执行删除命令释放旧资源，再执行创建命令创建新资源
  • 提高执行效率：批量处理相同类型的命令，减少状态切换
  • 线程安全：创建队列和删除队列分别加锁，支持多线程并发投递

11.2 队列结构

+-------------------+ +-------------------+ | 创建队列 | | 删除队列 | | (CreateQueue) | | (DestroyQueue) | +-------------------+ +-------------------+ | CreateBuffer | | DestroyBuffer | | CreateTexture | | DestroyTexture | | CreateInputLayout | | DestroyInputLayout| | UpdateBuffer | | | | UploadTexture | | | | BeginFrame | | | | EndFrame | | | | ExecuteDelegate | | | +-------------------+ +-------------------+ | | +----------+----------+ | v +-------------------+ | 执行顺序 | | 1. 先执行删除队列 | | 2. 再执行创建队列 | +-------------------+

11.3 命令分类

队列 命令类型 说明
创建队列 CreateBuffer 创建缓冲区
CreateTexture 创建纹理
CreateEmptyTexture 创建空纹理
UpdateBuffer 更新缓冲区数据
UploadTexture 上传纹理数据
UpdateTexture 局部更新纹理
CreateInputLayout 创建输入布局
BeginFrame/EndFrame/ExecuteDelegate 帧命令/委托命令
删除队列 DestroyBuffer 销毁缓冲区
DestroyTexture 销毁纹理
DestroyInputLayout 销毁输入布局

11.4 执行流程

void RenderThread::DrainOnce() {
    commandBuffer_.clear();
    
    // 1. 先取出删除队列的所有命令
    queue_.PopAllDestroyCommands(commandBuffer_);
    
    // 2. 再取出创建队列的所有命令
    queue_.PopAllCreateCommands(commandBuffer_);

    // 3. 按顺序执行（先删除后创建）
    for (const auto& cmd : commandBuffer_) {
        executor_->Execute(cmd);
    }
}

12. 上下文交接机制

12.1 设计目的

OpenGL 上下文是线程绑定的，一个上下文只能在一个线程中使用。上下文交接机制允许将 OpenGL 上下文从主线程转移到渲染线程。

12.2 交接流程

主线程 (ApplicationModule) 渲染线程 (RenderThread) | | | 1. Startup() | | - 创建窗口 | | - glfwCreateWindow() | | - 创建 OpenGL 上下文 | | - glfwMakeContextCurrent() | | | | 2. InitializeFromApplication() | | - 获取上下文指针 | | - glContext = GetGLContext()| | | | 3. ReleaseGLContext() | | - glfwMakeContextCurrent() | | - 释放上下文 | | | | | 4. ThreadMain() | | - glfwMakeContextCurrent() | | - 获取上下文 | | - 执行 OpenGL API

12.3 关键代码

// ApplicationModule::InitGlfw()
glfwMakeContextCurrent(window_);
glContext_ = window_;  // GLFW 的上下文与窗口绑定

// ApplicationModule::ReleaseGLContext()
glfwMakeContextCurrent(nullptr);  // 释放当前线程的上下文

// RenderThread::ThreadMain()
if (glWindow_) {
    glfwMakeContextCurrent(glWindow_);  // 在渲染线程获取上下文
}

// RenderModule::InitializeFromApplication()
void* glContext = app.GetGLContext();
GLFWwindow* window = app.GetWindow();
renderThread_->SetGLContext(window, glContext);
app.ReleaseGLContext();  // 主线程释放上下文
renderThread_->Start();  // 渲染线程获取上下文

12.4 注意事项

重要提示：
  • 上下文交接后，主线程不能再使用任何 OpenGL 调用
  • 渲染线程独占 OpenGL 上下文，所有图形 API 调用必须在渲染线程执行
  • 窗口销毁前必须先停止渲染线程，确保上下文已释放

14. 总结

Render 模块采用前端命令代理 + 专职渲染线程 + 双队列架构 + 上下文交接的架构，实现了：

  • 跨图形 API 抽象：上层不直接依赖 OpenGL 具体类型
  • 命令代理执行：所有渲染操作通过命令队列提交
  • 双队列设计：创建队列和删除队列分离，避免资源悬挂引用
  • 线程安全：图形 API 调用只在渲染线程执行
  • 上下文交接：支持 OpenGL 上下文从主线程转移到渲染线程
  • 灵活扩展：支持未来多后端、多队列、并行翻译
  • 快速验证：委托后门支持最短路径功能验证
